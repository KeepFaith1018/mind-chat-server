# 联网搜索生产配置说明（Tavily 主通道 + DuckDuckGo 兜底）

## 1. 当前实现策略

后端搜索链路已调整为：

1. 优先使用 Tavily（生产常用搜索 API）
2. Tavily 不可用时自动降级到 DuckDuckGo
3. 两条链路均有日志，便于观测

对应代码：

- `src/modules/search/search.service.ts`

---

## 2. 环境变量配置

在后端环境变量中增加：

```env
# 主搜索（推荐）
TAVILY_API_KEY=tvly-xxxxxxxxxxxxxxxx

# 可选：覆盖 Tavily 默认地址
WEB_SEARCH_ENDPOINT_TAVILY=https://api.tavily.com/search

# 可选：覆盖 DuckDuckGo 兜底地址
WEB_SEARCH_ENDPOINT_DDG=https://api.duckduckgo.com/
```

说明：

1. 有 `TAVILY_API_KEY` 才会启用主通道 Tavily
2. 没有 `TAVILY_API_KEY` 时只会走 DuckDuckGo

---

## 3. 运行时日志观测

成功日志：

1. `WebSearch.Completed`（`provider: tavily`）
2. `WebSearch.Completed`（`provider: duckduckgo`）

异常日志：

1. `WebSearch.TavilyNonOkResponse`
2. `WebSearch.TavilyFailed`
3. `WebSearch.DDGNonOkResponse`
4. `WebSearch.DDGFailed`

聊天链路侧：

1. `AIChat.WebSearchReady`（会看到 query 与 resultCount）

---

## 4. 联调验证步骤

1. 确认模型配置 `supports_web_search=true`
2. 前端请求带 `capabilities.webSearch=true`
3. 发起“最新/实时”类问题
4. 检查日志是否出现：
   - `WebSearch.Completed`（优先 tavily）
   - `AIChat.WebSearchReady` 且 `resultCount > 0`
5. 前端检查 `tool_result.items.length > 0`

---

## 5. 生产建议

1. 将 `TAVILY_API_KEY` 放入密钥管理（不要写死仓库）
2. 为搜索请求设置单独告警（连续失败率）
3. Nginx 出口网络放通 Tavily 域名
4. 可按业务量增加搜索缓存，减少成本和延迟
