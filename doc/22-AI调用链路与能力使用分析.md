# AI 调用链路与能力使用分析

## 1. 分析范围

本文聚焦当前项目中与 AI 对话直接相关的后端实现，覆盖：

1. AI 调用主链路
2. 模型能力判定与开关生效逻辑
3. Prompt 组装与上下文注入
4. 流式结果返回（SSE）机制
5. 计费、配额、安全在对话链路中的位置

核心代码位置：

- `src/modules/chat/chat.service.ts`
- `src/modules/ai/ai.service.ts`
- `src/modules/models/models.service.ts`
- `src/modules/chat/services/prompt.service.ts`
- `src/common/utils/sse.utils.ts`

---

## 2. AI 调用主链路（ChatService）

入口是 `POST /v1/chat`，在 `ChatService.chat()` 中按如下顺序执行：

1. 输入安全检查  
   - `safetyService.assertSafeInput(dto.messages)`
2. 配额检查  
   - `quotaService.checkQuota(userId)`
3. 会话解析  
   - `getOrCreateConversation(userId, dto)`
4. 模型选择  
   - `modelsService.resolveModelForChat(dto.model || conversation.modelId)`
5. 能力计算  
   - `modelsService.resolveCapabilities(...)`
6. 能力约束校验（stream/fileQa/webSearch）
7. 同步会话模型  
   - `syncConversationModel(conversation.id, selectedModel.modelKey)`
8. 可选联网检索  
   - `searchService.search(query)`，并回传 `tool_result`
9. Prompt 组装  
   - `promptService.buildMessages(...)`
10. 调用 AI 流式输出  
    - `aiService.streamChat(langChainMessages, options)`
11. 逐 chunk 输出 SSE（thinking/content）
12. 结束后异步后处理  
    - 保存 assistant 消息
    - usage 计算落库
    - quota 计量

结论：当前架构是“**能力校验前置 + 流式主链路 + 结果后处理异步化**”。

## 3. 模型与能力使用逻辑

## 3.1 模型解析

`ModelsService.resolveModelForChat(preferredModelKey)` 逻辑：

1. 优先使用请求模型（`preferredModelKey`）且必须 `enabled=true`
2. 若不可用，回落到默认/最近更新的启用模型
3. 若无可用模型，抛 `SERVICE_UNAVAILABLE`

## 3.2 能力生效计算

`ModelsService.resolveCapabilities(model, requested, hasFiles)`：

- `webSearch` = 请求开关 && 模型支持
- `reasoning` = 请求开关 && 模型支持
- `fileQa` = (请求开关 || 上传了文件) && 模型支持
- `stream` = 模型支持

## 3.3 聊天时强约束

在 `chat.service.ts` 中：

1. 模型不支持 `stream` → 直接拒绝
2. 传了文件但不支持 `fileQa` → 直接拒绝
3. 请求 `webSearch` 但模型不支持 → 直接拒绝

这保证前端开关与后端真实能力一致，不会“假开关”。

## 4. Prompt 组装策略（PromptService）

`PromptService.buildMessages(...)` 的关键处理：

1. 历史截断：只保留最近 20 条消息
2. 文件上下文注入：从 `files` 表读取内容拼接成参考区块
3. 联网上下文注入：把 `webSearchContext` 也并入参考区块
4. 注入位置：最后一条用户消息前
5. 默认 System Prompt：若首条不是 system，则自动补一条
6. 能力相关附加 Prompt：
   - `reasoning=false` 时，追加“直接回答，不输出详细推理”
   - `webSearch=true` 时，追加“信息不足要明确说明并保守结论”

结论：当前是“**summary 未接入，recent + file/web context + system 约束**”的实用型策略。

---

## 5. AI 调用实现（AiService）

`AiService.createChatModel(options)` 当前特点：

1. 统一走 SiliconFlow OpenAI 兼容端点  
   - `baseURL = https://api.siliconflow.cn/v1`
2. 从环境变量读取 `SILICONFLOW_API_KEY`
3. 支持可选推理参数映射  
   - `reasoningEnabled` + `reasoningParamKey`
4. 默认开启流式 `streaming: true`
5. `streamChat()` 支持透传 `AbortSignal`

结论：当前是“**可配置模型 + 可配置推理开关 + 可中断流调用**”。

---

## 6. 结果返回机制（SSE）

SSE 由 `SSEUtils` 统一处理，返回格式：

```text
event: <event_name>
data: <payload>

```

关键事件：

1. `meta`：会话ID、模型、能力矩阵
2. `tool_result`：联网检索结果
3. `thinking`：推理增量
4. `content`：答案增量
5. `error`：错误信息并结束
6. `done`：`[DONE]` 并结束

响应头已针对 fetch+ReadableStream 兼容优化：

- `Content-Type: text/event-stream; charset=utf-8`
- `Cache-Control: no-cache, no-transform`
- `Connection: keep-alive`
- `X-Accel-Buffering: no`

---

## 7. 计费与配额挂载点

AI 输出完成后进入 `handlePostChat(...)`：

1. 保存 assistant 消息（含 reasoningContent）
2. `usageService.computeUsage(...)`  
   - 优先 provider usage，缺失时估算
3. `usageService.persistUsage(...)` 写 `usage_logs`
4. `quotaService.recordUsage(...)` 更新配额

这是“先响应、后结算”的结构，有利于首字速度与用户体验。

---

## 8. 安全控制挂载点

1. 输入安全：请求进入聊天前执行  
   - 阻断 prompt injection 关键词
2. 输出安全：每个 chunk 发送前执行  
   - 命中规则则替换安全文案并中断后续输出
3. 中断清理：监听 `res.close`，abort 上游流，防资源泄漏

---

## 9. 当前能力现状总结

## 9.1 已具备

1. 模型动态配置与能力矩阵
2. 聊天能力开关（webSearch/reasoning/fileQa）
3. 联网搜索注入与来源回传
4. 标准 SSE 流式输出
5. usage + quota 主链路落库
6. 输入/输出安全策略与接口限流

## 9.2 仍可增强

1. 会话摘要策略（summary 自动刷新）未接主链路
2. provider usage 解析可做更细厂商适配
3. 安全规则建议配置化与审计化
4. 联网检索可增加缓存与重排序

---

## 10. 给前端的对接要点（AI 相关）

1. 聊天必须按 SSE 事件驱动渲染（非一次性 JSON）
2. 以 `meta.capabilities` 作为“最终生效能力”展示依据
3. `tool_result` 作为引用来源组件数据源
4. 监听 `error` 和 `done` 做流结束态处理
5. 页面离开/停止生成时主动 abort 请求，后端会自动回收流
