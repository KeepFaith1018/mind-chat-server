## 2. AI 对话相关模块总览

当前和 AI 对话直接相关模块：

1. `chat`：对话主编排（入口、SSE、能力校验、后处理）
2. `ai`：底层模型调用（LangChain + SiliconFlow）
3. `models`：模型配置读取与能力矩阵计算
4. `search`：联网检索与上下文构造
5. `safety`：输入/输出安全策略
6. `usage`：token 统计与成本落库
7. `quota`：配额检查与扣减
8. `upload`：文件内容提取，供 fileQa 注入
9. `conversation`：会话生命周期管理
10. `auth`：登录态与用户身份（提供 userId）

核心入口文件：

- `src/modules/chat/chat.controller.ts`
- `src/modules/chat/chat.service.ts`

## 3. 功能清单（AI 对话域）

## 4. 模块职责与边界

## 4.1 ChatModule（编排层）

文件：

- `chat.controller.ts`
- `chat.service.ts`
- `services/prompt.service.ts`

职责：

1. 接收请求并协调所有子服务
2. 统一错误与 SSE 事件输出
3. 保持业务规则一致（能力校验、流程顺序）

不负责：

- 具体模型 SDK 细节
- 搜索引擎实现细节
- 配额与成本算法细节

## 4.2 AiModule（模型调用层）

文件：

- `ai.service.ts`

职责：

1. 创建 LangChain ChatOpenAI 实例
2. 透传模型名、推理参数、AbortSignal
3. 返回流式 chunk

## 4.3 ModelsModule（能力基准层）

文件：

- `models.service.ts`

职责：

1. 从 `model_configs` 读取配置
2. 解析可用模型与默认模型
3. 计算 requested/effective 能力

## 4.4 SearchModule（联网工具层）

文件：

- `search.service.ts`

职责：

1. 执行搜索请求
2. 清洗并结构化结果
3. 构造可注入 Prompt 的搜索上下文

## 4.5 SafetyModule（安全防护层）

文件：

- `safety.service.ts`

职责：

1. 输入风险词检测（前置拦截）
2. 输出片段拦截（流式发送前过滤）

## 4.6 Usage + Quota（计费治理层）

文件：

- `usage.service.ts`
- `quota.service.ts`

职责：

1. usage 计算与 `usage_logs` 落库
2. 配额检查与 token 扣减

## 5. 端到端时序流程（一次对话）

以下为 `POST /v1/chat` 的典型时序：

1. `ChatController.chat()` 调用 `ChatService.chat()`
2. `SafetyService.assertSafeInput()` 输入安全检查
3. `QuotaService.checkQuota()` 配额预检查
4. `getOrCreateConversation()` 会话获取/创建
5. `ModelsService.resolveModelForChat()` 选模型
6. `ModelsService.resolveCapabilities()` 算 effective 能力
7. 能力校验（stream/fileQa/webSearch）
8. `syncConversationModel()` 同步会话模型
9. 若启用 webSearch：`SearchService.search()`
10. SSE 发 `meta`（会话ID、模型、生效能力）
11. SSE 发 `tool_result`（可选，搜索结果）
12. `PromptService.buildMessages()` 组装上下文
13. `AiService.streamChat()` 启动模型流
14. 循环消费 chunk：
    - reasoning -> SSE `thinking`
    - content -> `SafetyService.filterOutputChunk()` -> SSE `content`
15. SSE 发 `done`
16. 异步 `handlePostChat()`：
    - 保存 assistant 消息
    - UsageService 计算并落库
    - QuotaService 扣减配额

异常分支：

- 任意阶段异常 -> SSE `error` + 日志 `AIChat.Failed`

中断分支：

- 客户端断开 -> `res.close` -> Abort 上游模型流

## 6. 能力开关生效机制

输入来源：

- 前端 `dto.capabilities` 请求开关
- 当前模型支持矩阵（`model_configs`）
- 请求上下文（是否上传文件）

计算规则（简化）：

1. `webSearch = requested.webSearch && model.supportsWebSearch`
2. `reasoning = requested.reasoning && model.supportsReasoning`
3. `fileQa = (requested.fileQa || hasFiles) && model.supportsFileQa`
4. `stream = model.supportsStream`

结果分两层：

1. `requested`：前端想要的能力
2. `effective`：后端实际生效能力（最终执行依据）

`GET /v1/chat/capabilities` 可提前预览这个结果。

## 7. SSE 结果返回模型

核心事件：

1. `meta`
2. `tool_result`
3. `thinking`
4. `content`
5. `error`
6. `done`

协议关键点：

1. 标准 SSE 帧，双换行分隔
2. Header 已适配 fetch+ReadableStream
3. 错误通过流内 `error` 事件返回，避免中途 JSON 破坏解析

- `doc/21-SSE协议契约-前后端联调规范.md`

## 8. 依赖关系图（逻辑）

```text
ChatController
  -> ChatService
      -> SafetyService (input/output)
      -> QuotaService (check/record)
      -> ModelsService (resolve model/capabilities)
      -> SearchService (optional web search)
      -> PromptService (context assembly)
      -> AiService (LLM stream)
      -> UsageService (usage logs/cost)
      -> PrismaService (conversation/messages persistence)
      -> SSEUtils (stream output)
```

说明：

- ChatService 是编排中心，其他服务是能力插件。
- 这种结构便于扩展新工具（例如 rerank、function calling）。

## 9. 可观测性与日志

当前 AI 对话链路日志点：

1. `AIChat.Start`
2. `AIChat.ModelResolved`
3. `AIChat.WebSearchReady`
4. `AIChat.PromptBuilt`
5. `AIChat.StreamStarted`
6. `AIChat.StreamCompleted`
7. `AIChat.PostProcessCompleted`
8. `AIChat.OutputBlocked`
9. `AIChat.PostProcessFailed`
10. `AIChat.Failed`

排障建议顺序：

1. 先看 `AIChat.Start` 是否进入
2. 再看 `ModelResolved`/`WebSearchReady` 判定能力
3. 看 `StreamStarted` 到 `StreamCompleted` 是否完整
4. 看 `PostProcessCompleted` 判断落库与计费是否成功

## 11. 快速理解建议（新同学上手）

建议阅读顺序：

1. `chat.controller.ts`（入口）
2. `chat.service.ts`（编排主干）
3. `models.service.ts`（能力生效）
4. `prompt.service.ts`（上下文构造）
5. `ai.service.ts`（模型流调用）
6. `sse.utils.ts`（输出协议）
7. `usage/quota/safety/search`（横切能力）

这样能最快建立“请求如何穿过系统并返回流”的完整心智模型。
