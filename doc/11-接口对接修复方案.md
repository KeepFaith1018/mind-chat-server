# 接口对接修复方案

本文档记录了在测试阶段发现的接口对接问题，并提出了相应的修复方案。

## 1. 问题记录

### 1.1. 前端交互反馈缺失

**现象**：

- 用户注册/登录操作后，界面无任何提示（如成功/失败消息）。
- 注册成功后，期望自动切换到登录 Tab，但目前直接跳转或无反应。

### 1.2. 接口错误处理不当

**现象**：

- 注册接口返回 400 错误时，前端未能正确解析错误信息，导致控制台报错或无提示。
- 错误响应结构如下：
  ```json
  {
    "url": "http://localhost:3000/api/auth/register",
    "statusCode": 400,
    "statusMessage": "Server Error",
    "message": "该邮箱已被注册",
    "stack": [...]
  }
  ```
- 当前前端直接捕获了原始 Error 对象，未提取 `message` 字段展示给用户。

### 1.3. BigInt 序列化错误

**现象**：

- 调用 `/api/auth/me` 接口时，返回 500 错误：
  ```json
  {
    "statusCode": 500,
    "statusMessage": "Server Error",
    "message": "Do not know how to serialize a BigInt",
    ...
  }
  ```

**原因**：

- `UsageQuota` 表中的 `totalTokenUsage` 字段类型为 `BigInt`。
- JavaScript 的 `JSON.stringify` 默认不支持 `BigInt` 类型，导致 Nitro 在序列化响应时报错。

## 2. 修复方案

### 2.1. 引入全局反馈机制 (Toast)

使用 Nuxt UI 的 `useToast` 组件，为所有关键操作添加反馈。

- **成功提示**：操作成功时弹出绿色 Toast。
- **错误提示**：捕获异常时弹出红色 Toast，内容为后端返回的 `message`。

### 2.2. 优化错误解析逻辑

封装一个通用的错误解析函数 `getErrorMessage(error)`，用于从 Nuxt/Ofetch 的错误对象中提取人类可读的信息。

```typescript
// 示例逻辑
export function getErrorMessage(error: any): string {
  // 优先提取后端返回的 message
  if (error.response?._data?.message) {
    return error.response._data.message
  }
  // 其次提取 statusMessage
  if (error.response?.statusMessage) {
    return error.response.statusMessage
  }
  // 最后提取原生 message
  return error.message || '未知错误'
}
```

### 2.3. 调整注册流程

**现状**：注册成功 -> 自动登录 (Set-Cookie) -> 跳转 `/chat`。
**目标**：注册成功 -> 提示成功 -> 切换到登录 Tab -> 用户手动登录 (或自动填充)。

**注意**：由于后端 `register` 接口已经设置了 Auth Cookie，实际上用户注册后即为 **已登录状态**。为了满足"切换到登录 Tab"的需求，我们需要在前端做特殊处理：

1. 注册成功后，**不**立即跳转 `/chat`。
2. 显示"注册成功，请登录"提示。
3. 切换 Tab 到 `login`。
4. (可选) 自动填充邮箱。

_注：若用户刷新页面，由于 Cookie 存在，中间件可能会将其重定向到 `/chat`。这是一个由于"自动登录"机制与"手动登录"需求冲突导致的边缘情况，但在单页应用生命周期内是可控的。_

### 2.4. 解决 BigInt 序列化问题

**方案**：在全局或特定接口层面对 `BigInt` 进行处理，将其转换为字符串或数字。考虑到工程性，建议在 Prisma 客户端扩展或响应拦截器中统一处理，但最简单且无副作用的方式是：

**在 `server/utils/http.ts` 或 API 处理函数中，手动将 `BigInt` 字段转换为 `String`。**

由于 `/api/auth/me` 返回的是 `UsageQuota` 对象，我们需要在返回前进行转换：

```typescript
// server/api/auth/me.get.ts
const quota = {
  ...userQuota,
  totalTokenUsage: userQuota.totalTokenUsage.toString() // BigInt -> String
}
```

同时，为了防止后端内部错误泄露给前端（如 Stack Trace），应确保生产环境配置了正确的错误处理策略（Nuxt 默认在生产环境隐藏 Stack，但 message 可能会透传）。建议：

- **开发环境**：保留详细错误以便调试。
- **生产环境**：统一捕获未知异常，返回通用错误提示 "Internal Server Error"，并记录日志。

## 3. 实施步骤

1.  **Step 1: 创建错误处理工具**
    - 在 `app/utils/error.ts` 中实现 `getErrorMessage`。

2.  **Step 2: 改造 `login.vue`**
    - 引入 `const toast = useToast()`。
    - 在 `onLoginSubmit` 和 `onRegisterSubmit` 中添加 `try/catch` 块。
    - `catch` 块中调用 `toast.add({ title: '错误', description: getErrorMessage(err), color: 'red' })`。
    - `onRegisterSubmit` 成功后：
      - `toast.add({ title: '注册成功', color: 'green' })`
      - `activeTab.value = 'login'` (切换 Tab)
      - `loginState.email = registerState.email` (自动填充)

3.  **Step 3: 修复 BigInt 问题**
    - 修改 `server/api/auth/me.get.ts`，在返回前将 `totalTokenUsage` 转换为字符串。
    - 检查其他涉及 `UsageQuota` 的接口（如 `register`），确保同样处理。

4.  **Step 4: 验证**
    - 验证注册流程交互。
    - 验证 `/api/auth/me` 接口是否正常返回 JSON。

## 4. 待确认事项

- **自动登录 vs 手动登录**：当前后端注册接口默认包含自动登录逻辑（Set-Cookie）。前端仅做 UI 上的"伪"手动登录流程（切换 Tab），用户实际已持有 Token。这不会影响功能，但逻辑上略有冗余。是否需要后端调整 `register` 接口不返回 Cookie？
  - **建议**：暂时保持后端不变，前端仅做 UI 引导，减少后端改动风险。
