# 后端认证模块接口设计与实现方案

## 1. 概述 (Overview)

本文档详细描述了 MindChat 项目后端认证模块的接口定义与技术实现方案。基于 Nuxt 4 + Nitro + Prisma 技术栈，采用双 Token (Access Token + Refresh Token) 机制并配合 HttpOnly Cookie 进行安全存储，同时集成 GitHub 与 Google 第三方登录。

---

## 2. 技术方案 (Technical Architecture)

### 2.1 认证机制 (Authentication Mechanism)

采用 **双 Token 机制** 以平衡安全性与用户体验：

- **Access Token**: 短期有效（如 15 分钟），用于访问受保护的 API 资源。
- **Refresh Token**: 长期有效（如 7 天），用于在 Access Token 过期后获取新的 Token 对。

### 2.2 Token 存储与传输 (Storage & Transmission)

放弃 LocalStorage 存储方案，改用 **HttpOnly Cookie**，以防止 XSS 攻击窃取 Token。

- **Cookie 配置**:
  - `httpOnly: true` (禁止 JS 访问)
  - `secure: process.env.NODE_ENV === 'production'` (仅 HTTPS)
  - `sameSite: 'lax'` (防止 CSRF，同时允许正常跳转)
  - `maxAge`: 对应 Token 有效期

### 2.3 密码安全

使用 `bcrypt` (或 `argon2`) 对用户密码进行哈希存储，绝不明文保存密码。

### 2.4 用户初始化

用户注册或首次第三方登录成功后，系统自动触发 **配额初始化** 流程，在 `UsageQuota` 表中创建默认记录。

---

## 3. 数据库模型回顾 (Database Schema Review)

基于现有 `prisma/schema.prisma`：

- **User**: 核心用户表 (email, password_hash, name, avatar_url)。
- **Account**: 第三方登录关联表 (provider, provider_id, access_token)。
- **UsageQuota**: 用户用量配额表 (daily_token_limit)。

## 4. 接口定义 (API Specification)

### 4.1 用户注册

- **Endpoint**: `POST /api/auth/register`
- **描述**: 邮箱/密码注册，创建用户并初始化配额。
- **Request Body**:
  ```json
  {
    "email": "user@example.com",
    "password": "strongPassword123",
    "name": "MyName" // 可选
  }
  ```
- **Response**: 200 OK
  ```json
  {
    "code": 200,
    "message": "注册成功",
    "data": {
      "user": { "id": "...", "email": "...", "name": "..." }
    }
  }
  ```
- **Side Effects**:
  - 设置 `access_token` 和 `refresh_token` Cookies。
  - 创建 `UsageQuota` 记录。

### 4.2 用户登录

- **Endpoint**: `POST /api/auth/login`
- **描述**: 邮箱/密码登录。
- **Request Body**:
  ```json
  {
    "email": "user@example.com",
    "password": "strongPassword123"
  }
  ```
- **Response**: 200 OK (User info) + Cookies

### 4.3 获取当前用户信息

- **Endpoint**: `GET /api/auth/me`
- **描述**: 根据 Cookie 中的 Access Token 获取当前用户信息。
- **Headers**: 自动携带 Cookie。
- **Response**: 200 OK (User info + Quota info)

### 4.4 刷新 Token

- **Endpoint**: `POST /api/auth/refresh` (或 `GET`)
- **描述**: 当 Access Token 过期时，使用 Refresh Token 获取新的 Token 对。
- **Logic**:
  1. 验证 Cookie 中的 `refresh_token`。
  2. 签发新的 `access_token` 和 `refresh_token`。
  3. 更新 Cookies。

### 4.5 退出登录

- **Endpoint**: `POST /api/auth/logout`
- **描述**: 清除服务端 Session (可选) 并清除客户端 Cookies。
- **Response**: 200 OK

### 4.6 GitHub 登录

- **Endpoint**: `GET /api/auth/github`
- **描述**: 重定向至 GitHub OAuth 授权页面。
- **Params**: `redirect` (登录成功后的回跳地址)。

### 4.7 GitHub 回调

- **Endpoint**: `GET /api/auth/github/callback`
- **描述**: 处理 GitHub 回调，创建/关联账号，设置 Cookies，重定向回前端。

### 4.8 Google 登录

- **Endpoint**: `GET /api/auth/google`
- **描述**: 重定向至 Google OAuth 授权页面。

### 4.9 Google 回调

- **Endpoint**: `GET /api/auth/google/callback`
- **描述**: 处理 Google 回调。

## 5. 实现方案细节 (Implementation Details)

### 5.1 工具类封装 (`server/utils`)

#### `jwt.ts` (更新)

增加 Refresh Token 的签发与验证逻辑。

```typescript
export const generateTokens = (user: User) => {
  const accessToken = sign({ userId: user.id }, secret, { expiresIn: '15m' })
  const refreshToken = sign({ userId: user.id, type: 'refresh' }, secret, { expiresIn: '7d' })
  return { accessToken, refreshToken }
}
```

#### `cookie.ts` (新增)

统一管理 Cookie 的设置与清除。

```typescript
export const setAuthCookies = (event: H3Event, tokens: { accessToken: string, refreshToken: string }) => {
  setCookie(event, 'access_token', tokens.accessToken, { httpOnly: true, ... });
  setCookie(event, 'refresh_token', tokens.refreshToken, { httpOnly: true, ... });
}
```

### 5.2 环境变量配置 (.env)

```env
# Database
DATABASE_URL="postgresql://..."

# JWT
JWT_ACCESS_SECRET="complex_secret_access"
JWT_REFRESH_SECRET="complex_secret_refresh"

# OAuth - GitHub
GITHUB_CLIENT_ID="..."
GITHUB_CLIENT_SECRET="..."

# OAuth - Google
GOOGLE_CLIENT_ID="..."
GOOGLE_CLIENT_SECRET="..."
GOOGLE_REDIRECT_URI="http://localhost:3000/api/auth/google/callback"
```

### 5.3 默认配额策略

在 `register` 和 OAuth `callback` 逻辑中，当检测到是新用户创建时：

```typescript
await prisma.usageQuota.create({
  data: {
    userId: user.id,
    dailyTokenLimit: 500000 // 默认 50万 Token
  }
})
```

## 6. 异常处理 (Error Handling)

- **401 Unauthorized**: Token 无效或过期。
- **403 Forbidden**: 无权限。
- **400 Bad Request**: 参数错误或邮箱已存在。
- **500 Internal Server Error**: 数据库连接失败等。

统一使用 `createError({ statusCode, message })` 抛出异常，由 Nitro 全局捕获处理。
