# 认证模块前后端对接方案

本文档详细描述了 MindChat 项目认证模块的前后端对接计划。基于后端已实现的 **HttpOnly Cookie + 双 Token** 机制，前端（Nuxt 4）需要进行相应的适配，不再手动管理 Token，而是通过 API 交互和状态同步来维持用户会话。

## 1. 核心架构变更

| 特性           | 旧方案 (Mock/Client Side)         | 新方案 (Server Side Auth)                         |
| :------------- | :-------------------------------- | :------------------------------------------------ |
| **Token 存储** | 前端 `useCookie` / `localStorage` | **HttpOnly Cookie** (前端不可读，自动携带)        |
| **状态维持**   | 检查 Token 是否存在               | 调用 `/api/auth/me` 获取用户信息                  |
| **CSRF 防护**  | 依赖 Token                        | 依赖 Cookie 属性 (SameSite=Lax, HttpOnly)         |
| **Token 刷新** | 前端定时器或拦截器手动刷新        | 拦截 401 响应 -> 调用 `/api/auth/refresh` -> 重试 |
| **类型共享**   | 前后端各自定义                    | 统一维护在 `share/types.ts`                       |

## 2. 接口清单

前端需要对接以下后端接口：

| 接口路径             | 方法 | 描述         | 参数                           | 响应                                    |
| :------------------- | :--- | :----------- | :----------------------------- | :-------------------------------------- |
| `/api/auth/register` | POST | 用户注册     | `{ email, password, name }`    | User 对象 (自动 Set-Cookie)             |
| `/api/auth/login`    | POST | 用户登录     | `{ email, password }`          | User 对象 (自动 Set-Cookie)             |
| `/api/auth/me`       | GET  | 获取当前用户 | 无 (依赖 Cookie)               | `{ user: User, quota: UsageQuota }`     |
| `/api/auth/logout`   | POST | 退出登录     | 无                             | 成功信息 (自动 Clear-Cookie)            |
| `/api/auth/refresh`  | POST | 刷新 Token   | 无 (依赖 Refresh Token Cookie) | 成功信息 (自动更新 Access Token Cookie) |
| `/api/auth/github`   | GET  | GitHub 登录  | 无                             | **浏览器重定向**                        |
| `/api/auth/google`   | GET  | Google 登录  | 无                             | **浏览器重定向**                        |

## 3. 前端改造计划

### 3.1. Store 改造 (`app/stores/useAuthStore.ts`)

**现状**：存储了 Mock 的 Token 和 User 信息。
**修改目标**：

1.  **移除** `accessToken` 和 `refreshToken` 的定义（前端无法直接访问）。
2.  保留 `user` 状态，但不再持久化到 `localStorage` (或者仅作为缓存)，推荐使用 `useState` (Nuxt 内置) 以支持 SSR 数据同步。
3.  新增 `fetchUser` action，用于初始化和更新用户信息。

```typescript
// 伪代码示例
export const useAuthStore = defineStore('auth', () => {
  const user = useState<User | null>('auth-user', () => null)
  const isLoggedIn = computed(() => !!user.value)

  // 初始化用户 (在 App 启动或 Layout 中调用)
  async function fetchUser() {
    try {
      const { data } = await useFetch('/api/auth/me')
      if (data.value?.code === 200) {
        user.value = data.value.data.user
      } else {
        user.value = null
      }
    } catch (e) {
      user.value = null
    }
  }

  async function login(credentials) {
    await $fetch('/api/auth/login', { method: 'POST', body: credentials })
    await fetchUser() // 登录成功后拉取用户信息
  }

  async function logout() {
    await $fetch('/api/auth/logout', { method: 'POST' })
    user.value = null
    navigateTo('/login')
  }

  return { user, isLoggedIn, fetchUser, login, logout }
})
```

### 3.2. 全局初始化插件 (`app/plugins/auth.ts`)

为了解决 SSR 期间的鉴权状态同步问题，我们需要创建一个 Nuxt 插件，在应用启动时（服务端和客户端）尝试获取用户信息。

```typescript
export default defineNuxtPlugin(async () => {
  const authStore = useAuthStore()
  // 仅在 user 为空时请求，避免重复请求
  if (!authStore.user) {
    await authStore.fetchUser()
  }
})
```

### 3.3. 登录页面适配 (`app/pages/login.vue`)

1.  **表单提交**：替换 `authStore.login/register` 为真实的 API 调用。
2.  **第三方登录**：
    - 移除 `setTimeout` 模拟代码。
    - 直接使用 `window.location.href = '/api/auth/github'` 进行跳转。
    - 由于是全页跳转，无需处理回调逻辑（后端回调后会重定向回 `/`）。

### 3.4. API 请求封装与 Token 刷新

建议在 `composables/useHttp.ts` 或类似地方封装 `$fetch`，统一处理 401 错误。

```typescript
// 拦截器逻辑示意
async function onResponseError({ response }) {
  if (response.status === 401) {
    // 尝试刷新 Token
    try {
      await $fetch('/api/auth/refresh', { method: 'POST' })
      // 刷新成功，重试原请求 (需要复杂的重试逻辑，或者简单地让用户刷新页面/重新操作)
      // 最简单的 MVP 方案：刷新成功后不做操作，下次请求自然会带上新 Cookie；
      // 如果刷新也失败，则执行登出
    } catch (e) {
      // 刷新失败，强制登出
      const authStore = useAuthStore()
      authStore.logout()
    }
  }
}
```

## 5. 注意事项

- **TypeScript 类型定义**：为了保证前后端类型一致性，所有公共类型定义（如 `User`, `ApiResponse`）都统一存放在 `/share/types.ts` 中。前端和后端都应引用该文件。
  - 前端引用：`import type { User, IApiResponse } from '~/share/types'`
  - 后端引用：`import type { User, IApiResponse } from '../../../share/types'` (根据相对路径调整)
- **SSR Context**：在服务端渲染期间，Nuxt 会自动将请求头的 Cookie 转发给后端 API（使用 `useFetch` 时），这是 SSR 鉴权成功的关键。
- **开发环境跨域**：Nuxt Server 代理了 `/api` 请求，因此在开发环境下不存在跨域问题。
